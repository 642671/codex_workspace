## 3. 模块定位

### 3.1 存储管理模块在 TNAS 体系中的角色

在当前测试语境里，存储管理模块不是普通配置页，而是 TNAS 存储主链路的核心承载区域。  
它直接关系到：

- 磁盘是否被识别
- 存储池是否被正确建立
- 卷是否可创建、可使用、可扩展
- 容量、状态、可用性是否真实可信
- 后续文件服务、数据访问、修复恢复是否有基础支撑

如果这个模块判断错了，后续很多页面、功能、接口现象都会被连带影响。

### 3.2 为什么它是高频主模块

它之所以是高频主模块，不是因为页面多，而是因为它天然位于测试主链路中：

- 初始化之后通常首先要确认磁盘和存储对象
- 日常回归会频繁涉及容量、状态、扩容、删除、修复
- 很多异常、投诉、风险最终都会回到“存储对象是否真的正常”

因此它不是边缘模块，而是测试中高频反复触达的核心模块。

### 3.3 为什么它是高风险模块

它之所以高风险，主要因为这里的错误通常不只是页面问题，而会直接影响：

- 数据可用性
- 对象关系
- 状态流转
- 容量正确性
- 用户对数据安全的判断
- 版本准出风险

尤其是以下操作天然高风险：

- 删除
- 扩容
- 迁移
- 修复
- 异常恢复

### 3.4 为什么它要优先进入模块层

当前共享层已经把测试视角、工作流、证据链和输出规范稳定下来了。  
要让共享层第一次真正落地，就必须选一个：

- 业务链路清晰
- 风险高
- 高频出现
- 能明显体现“页面现象不能单独下结论”的模块

存储管理模块最适合作为第一个下沉模块。

## 4. 当前模块下的第一层对象范围

当前阶段，这个模块先不无限展开子对象，而是先建立**第一层对象地图**。  
当前已确认应纳入存储管理模块入口视角的第一层对象包括：

- 磁盘
- 存储池
- 卷
- 虚拟磁盘
- USB设备
- 热备盘
- Hyper Cache

### 4.1 对象地图的当前理解

当前可以先按下面的测试理解关系去看：

磁盘  
→ 存储池  
→ 卷  
→ 上层使用与容量呈现

同时还有几类扩展对象需要纳入观察，但当前不作为第一优先固化对象：

- 虚拟磁盘
- USB设备
- 热备盘
- Hyper Cache

### 4.2 当前优先级

当前模块层下沉优先级先固定为：

- **第一优先固化：存储池 + 卷**
- **第二优先固化：磁盘**

原因是：

- 存储池和卷更直接决定对象状态、容量、可用性、操作结果
- 磁盘虽然是底层对象，但在当前入口阶段更多是作为前置对象和风险来源
- 虚拟磁盘、USB设备、热备盘、Hyper Cache 当前先作为扩展对象保留入口，不在这一版里展开到细规则层

## 5. 核心业务主链路（模块内）

从测试视角看，存储管理模块当前应先沿着以下主链路理解：

1. 识别磁盘
2. 判断磁盘是否满足创建条件
3. 创建存储池
4. 在存储池上创建卷
5. 查看状态 / 容量 / 使用情况
6. 继续进行编辑 / 删除 / 扩容 / 迁移 / 修复等操作
7. 观察操作前、中、后的对象关系、状态和展示是否正确

### 5.1 当前主链路重点不是实现细节，而是测试判断点

当前主链路下，测试上优先关心的不是后台具体如何实现，而是：

- 该出现的对象有没有出现
- 对象关系是不是对的
- 状态是不是能解释
- 容量是不是说得通
- 页面展示和接口返回是不是一致
- 失败、处理中、异常时是否给出了正确反馈

### 5.2 当前需要重点盯住的主链路场景

当前至少应把以下场景看作存储管理模块的高频主链路：

- 新磁盘识别
- 新建存储池
- 新建卷
- 查看对象状态
- 查看容量、已用、可用情况
- 删除存储池或卷
- 扩容
- 迁移
- 修复

## 6. 当前已确认的规则入口

当前这一版不强行写死完整状态值集合，而是先固化**规则入口**和**判断方向**。

### 6.1 存储池 / 卷状态口径入口

当前已确认：

- 存储池和卷是当前模块层最优先固化的状态判断对象
- 状态判断不能只看单个页面文案
- 状态必须结合真实对象、真实操作阶段、真实接口返回来理解
- 过程态必须纳入判断，不能只看最终态

当前已经明确应重点关注的过程态方向包括：

- `分析中`
- `加载中`
- `处理中`

说明：

- 当前这里只固化“过程态应纳入判断”这一原则
- 暂不把所有状态值和完整迁移规则写死
- 完整状态集合仍属于 `待补充 / 待确认`

### 6.2 哪些状态通常应一致

当前已确认，以下内容通常应保持一致，或者至少能相互解释：

- 同一个对象在列表、详情、卡片等不同展示位的状态
- 存储池状态与卷可操作性之间的关系
- 对象状态与容量展示之间的关系
- 页面状态与接口返回状态
- 页面展示对象关系与后台真实对象关系

### 6.3 哪些状态允许不一致

当前已确认，以下场景下允许出现**暂时不一致**，但必须是可解释的不一致：

- 过程态下的短暂刷新不同步
- 页面轮询与接口返回存在先后顺序差异
- 上下游对象状态存在阶段性传递延迟
- 容量统计、使用量统计在刷新前后存在短暂差异

需要强调：

- 当前只确认“可能存在短暂且可解释的不一致”
- 没有证据前，不能擅自写死允许时长、允许次数或允许范围
- 只要不一致不可解释、长期不恢复、前后台口径冲突，就不能轻易放过

### 6.4 哪些规则影响页面判断

当前已确认会直接影响页面判断的规则入口包括：

- 对象是否存在
- 对象状态是否符合当前操作阶段
- 容量、已用、可用是否说得通
- 操作入口是否该显示、该禁用、该隐藏
- 删除、迁移、修复、扩容时的过程展示是否合理
- 异常、失败时是否有可理解的提示

### 6.5 哪些规则影响接口判断

当前已确认会直接影响接口判断的规则入口包括：

- 接口返回的对象状态是否与页面一致
- 接口返回的容量字段是否与页面口径一致
- 接口返回的对象关系是否正确
- 过程态接口是否能反映操作阶段变化
- 失败场景接口是否返回了可用于定位问题的信息

### 6.6 哪些规则影响缺陷判断

当前已确认会直接影响缺陷判断的规则入口包括：

- 同一对象前后端状态冲突
- 对象关系错误
- 容量计算明显异常
- 删除、迁移、修复、扩容后结果不符合实际现象
- 过程态卡死、长时间不结束、结束后结果错误
- 页面提示、接口返回、日志信息彼此无法解释

## 7. 当前已确认的展示与判定口径

### 7.1 页面展示不能只看前端

当前已确认：

- 页面现象只能作为入口，不能直接作为结论
- 存储管理模块尤其不能只凭页面字面文案下结论
- 必须结合真实状态、真实接口返回、必要日志或系统信息判断

### 7.2 前后端一致性是重点

当前模块判断里，前后端一致性属于核心检查方向，包括：

- 状态一致性
- 容量一致性
- 使用情况一致性
- 对象关系一致性
- 操作结果一致性

### 7.3 对象状态、容量、使用情况、关联关系是重点

当前已确认的重点判断维度包括：

- 对象状态
- 容量变化
- 已使用 / 可用空间
- 对象之间的关联关系

只要这几个维度里有一个解释不通，就不能轻易把现象当成正常。

### 7.4 过程态必须纳入判断

当前已确认：

- `分析中 / 加载中 / 处理中` 这类过程态不是附属信息
- 它们本身就是测试判断对象

测试时至少要关心：

- 过程态是否该出现时出现
- 过程态是否持续异常
- 过程态结束后是否进入正确结果
- 过程态期间页面、接口、对象关系是否还能相互解释

### 7.5 删除、迁移、修复、异常时的展示应重点关注

当前已确认，这几类场景的展示风险更高：

- 删除
- 迁移
- 修复
- 扩容
- 异常 / 失败

原因是这些场景更容易出现：

- 状态卡住
- 页面未刷新
- 后台已变化但前台未同步
- 入口未收敛
- 对象残留
- 容量或关系异常

## 8. 当前高风险测试点入口

这部分当前不展开成完整用例，只列出后续测试设计应优先进入的高风险方向。

### 8.1 状态流转

重点看：

- 状态是否按操作阶段变化
- 过程态是否合理
- 最终态是否与实际结果一致
- 失败后是否正确回退或保留异常状态

### 8.2 对象关系

重点看：

- 磁盘、存储池、卷之间关系是否正确
- 删除或异常后是否有残留对象
- 关联关系是否前后台一致

### 8.3 容量变化

重点看：

- 新建后容量是否正确
- 扩容后容量是否正确
- 删除后容量是否回收正确
- 页面、接口、系统信息中的容量口径是否相互解释

### 8.4 前后端不一致

重点看：

- 页面显示正常但接口真实状态异常
- 页面状态未更新但后台已变化
- 接口已成功但页面结果未落地
- 同一对象多处展示口径冲突

### 8.5 接口 / 页面口径不一致

重点看：

- 状态文案不一致
- 容量字段不一致
- 对象存在性不一致
- 操作结果不一致

### 8.6 异常 / 失败 / 边界场景

重点看：

- 前置条件不足时是否正确拦截
- 处理中断时是否正确反馈
- 失败后是否保留足够证据
- 边界容量、边界对象关系下是否出现异常展示

### 8.7 关键操作链路

当前应重点作为高风险入口的关键操作包括：

- 删除
- 迁移
- 修复
- 扩容

这几类操作后续更适合继续拆成专项规则或专项测试文档。

## 9. 当前模块的证据链入口

从测试工程师视角，存储管理模块的问题判断通常依赖以下证据源交叉验证：

- 页面现象
- Web 抓包
- 接口返回
- 日志 / 系统信息
- 文档 / 规则
- 缺陷复现步骤

### 9.1 页面现象

页面现象用于发现问题入口，重点包括：

- 对象是否出现
- 状态如何显示
- 容量如何显示
- 操作入口是否正确
- 操作后是否刷新到位

### 9.2 Web 抓包

Web 抓包用于验证：

- 页面操作到底触发了什么请求
- 请求成功后页面为什么没变
- 页面现象是否能被真实请求 / 响应解释

### 9.3 接口返回

接口返回用于验证：

- 对象真实状态
- 容量字段
- 对象关系
- 失败原因
- 过程态变化

### 9.4 日志 / 系统信息

日志和系统信息用于辅助定位：

- 页面与接口无法解释的问题
- 状态卡住的问题
- 修复、迁移、扩容等关键链路异常
- 容量或对象关系异常

### 9.5 文档 / 规则

文档和规则用于判断：

- 当前现象是否违背产品规则
- 页面或接口口径是否与既有规则冲突
- 缺陷是否有明确规则依据

### 9.6 缺陷复现步骤

复现步骤用于保证：

- 现象可追溯
- 判断有上下文
- 后续验证、关闭、回归可复用
