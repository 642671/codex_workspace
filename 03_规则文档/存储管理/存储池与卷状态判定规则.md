# 存储池与卷状态判定规则

## 1. 文档定位

这份文档是存储管理模块下的**共建模式规则文档**。  
它不是最终定稿规则库，也不是完整状态机文档，而是为了把“存储池 + 卷”这两个第一优先对象的状态判定，先搭成一个可以逐轮补充、逐轮确认、逐轮稳定的骨架。

当前这份文档只做三件事：

- 把已经确认的事实先收进来
- 把必须继续补的空缺显式列出来
- 把现在有方向、但还不能写死的规则单独隔离出来

后续这份文档默认按“共建模式”维护，不把未确认结论直接写成最终事实。

---

## 2. 已确认事实

### 2.1 对象优先级

- 在存储管理模块里，当前**第一优先固化对象**是 `存储池 + 卷`。
- `磁盘` 当前是第二优先固化对象。
- 存储池和卷之所以优先，是因为它们更直接决定对象状态、容量、可用性和操作结果。

### 2.2 状态判定的基本口径

- 存储池和卷是当前模块层最优先的状态判断对象。
- 状态判断不能只看单个页面文案。
- 状态判断必须结合真实对象、真实操作阶段、真实接口返回来理解。
- 过程态必须纳入判断，不能只看最终态。

### 2.3 当前已明确应纳入判断的过程态方向

- `分析中`
- `加载中`
- `处理中`

说明：

- 当前只能确认这些过程态方向需要纳入判断。
- 还不能据此反推出完整状态集合或完整状态迁移规则。

### 2.4 哪些内容通常应保持一致

当前已确认，以下内容通常应保持一致，或者至少能相互解释：

- 同一个对象在列表、详情、卡片等不同展示位的状态
- 存储池状态与卷可操作性之间的关系
- 对象状态与容量展示之间的关系
- 页面状态与接口返回状态
- 页面展示对象关系与后台真实对象关系

### 2.5 哪些不一致当前可以先视为“可能合理”

当前已确认，以下场景可能出现**短暂且可解释的不一致**：

- 过程态下的短暂刷新不同步
- 页面轮询与接口返回存在先后顺序差异
- 上下游对象状态存在阶段性传递延迟
- 容量统计、使用量统计在刷新前后存在短暂差异

但当前只能确认“可能存在、且必须可解释”，不能写死为固定规则。

### 2.6 哪些判断方向已经明确会影响页面、接口、缺陷判断

当前已确认：

- 页面判断重点关注：对象是否存在、状态是否符合当前阶段、容量是否说得通、操作入口是否合理、异常提示是否可理解。
- 接口判断重点关注：对象状态、容量字段、对象关系、过程态变化、失败返回。
- 缺陷判断重点关注：前后端状态冲突、对象关系错误、容量明显异常、过程态卡死、操作结果与现象不一致。

### 2.7 当前高风险操作与状态判定直接相关

以下操作当前已确认属于高风险操作，且与状态判定直接相关：

- 删除
- 扩容
- 迁移
- 修复

这些操作更容易引发：

- 状态卡住
- 页面未刷新
- 后台已变化但前台未同步
- 对象残留
- 容量或对象关系异常

## 4. 待验证规则

这一部分只记录“现在有方向，但还不能写成已确认事实”的规则。

### 4.1 存储池与卷状态是否必须严格同步

当前方向：

- 存储池与卷状态多数情况下应一致或可解释
- 但在过程态、轮询刷新、操作传播阶段，可能允许短暂不同步

当前还不能写死的点：

- 哪些状态必须严格同步
- 哪些状态允许短暂不同步
- 允许不同步的具体条件、时长、边界

### 4.2 过程态是否应该阻止某些操作

当前方向：

- `分析中 / 加载中 / 处理中` 这类过程态应影响页面判断和操作入口判断

当前还不能写死的点：

- 哪些过程态必须禁止操作
- 哪些过程态允许部分操作
- 不同对象在过程态下的按钮禁用矩阵

### 4.3 容量变化是否应与状态变化强绑定

当前方向：

- 容量展示与对象状态应保持一致或可解释
- 扩容、删除、迁移、修复后，容量变化应进入重点判断范围

当前还不能写死的点：

- 哪些操作后容量应立即变化
- 哪些操作后容量允许延迟更新
- 页面容量与接口容量允许出现的短暂差异范围

### 4.4 存储池状态与卷可操作性之间是否存在固定映射

当前方向：

- 存储池状态会影响卷的可操作性

当前还不能写死的点：

- 哪些存储池状态对应卷必须不可操作
- 哪些状态下卷仍允许查看、删除、扩容或其它操作
- 是否存在存储池异常但卷仍部分可用的标准场景

### 4.5 删除、迁移、修复、扩容是否有统一的判定骨架

当前方向：

- 这几类高风险操作都应关注：操作前状态、过程态、结束态、前后端一致性、对象残留、容量变化

当前还不能写死的点：

- 各操作的标准过程态序列
- 各操作的结束成功 / 失败判定条件
- 各操作失败后的标准回退或保留状态

## 5. 风险提示

### 5.1 如果不先补稳状态口径，后续会直接影响缺陷判断

影响包括：

- 同样一个现象，不同轮次可能得出不同结论
- 页面问题、接口问题、真实业务问题容易混在一起
- 缺陷标题、严重程度、关闭口径容易失真

### 5.2 如果不补充过程态规则，后续很容易误判

影响包括：

- 把正常过程态误判成异常
- 把真实卡死误判成“只是慢”
- 无法判断按钮应不应该禁用

### 5.3 如果不补齐容量口径，后续很难稳定判断“是否异常”

影响包括：

- 页面容量异常无法和接口、日志、对象状态对齐
- 扩容、删除、迁移后的结果难以统一判定
- 容量问题容易在后续版本里反复 reopen

### 5.4 如果不先把存储池与卷关系补稳，多智能体后续拆分会失真

影响包括：

- 未来“状态判断型”“接口审查型”“缺陷分析型”智能体边界会重叠
- 规则没稳就分类，后续会反复返工
- 多智能体职责容易建立在错误底座上
