# TNAS 业务总览

## 1. TNAS 在当前测试语境里是什么

从当前测试语境看，TNAS 是铁威马的网络附属存储产品。  
它本质上是一个面向个人、家庭、办公小团队等场景使用的**私有云存储产品体系**，主要解决以下问题：

- 数据集中存储
- 数据管理
- 数据访问
- 数据共享
- 数据备份

对测试来说，TNAS 不能被理解成单个页面、单个接口或单个端的功能集合。  
它更应该被理解成一个围绕“存储对象、系统状态、访问管理、配置管理和异常处理”展开的业务体系。

## 2. 面向谁

当前已明确的典型用户和使用场景包括：

- 个人用户
- 家庭场景用户
- 办公小团队用户

这些用户使用 TNAS 的核心诉求，不是单点操作，而是围绕存储与访问形成长期使用行为。  
因此测试不能只验证单次点击结果，还需要关注持续使用中的状态变化和系统稳定表现。

## 3. 它解决什么问题

在当前业务理解下，TNAS 主要承载以下能力：

- 数据存储
- 数据管理
- 数据查看与访问
- 数据共享
- 数据备份
- 存储对象的创建、管理、变更和维护

在测试视角下，这意味着很多问题都不能只从界面行为判断。  
必须把对象关系、状态变化、容量变化和配置结果一起纳入判断。

## 4. 为什么它不是单一端产品

TNAS 不是单一端产品，而是一个多端协同的产品体系。  
当前已明确至少包括：

- 设备端
- Web 管理端
- 手机端

另外：

- PC 端或其他客户端能力当前可能涉及，这不是我要负责的模块，不需要考虑。

这意味着测试不能只在某一个端上判断“功能对不对”，还要关注：

- 同一对象在不同端的展示是否一致
- 同一状态在不同端的同步是否一致
- 页面现象是否能被真实接口结果解释
- 多端之间是否存在时序差异、状态延迟、展示口径不一致等问题

## 5. 多端协同特征

当前对 TNAS 的测试理解至少包含以下多端协同特征：

- 同一个业务对象可能同时被设备端、Web 端、手机端感知
- 同一个状态变化会影响多个页面、多个入口、多个展示位置
- 某些问题表面上出现在页面，但根因可能在接口、状态同步、对象关系或底层系统行为
- 文档、页面、抓包、日志之间必须能互相解释，不能各说各话

因此，后续分析 TNAS 时，默认要把以下一致性作为核心判断方向：

- 多端一致性
- 多对象一致性
- 多状态一致性
- 前后端一致性
- 文档与实现一致性

## 6. 当前理解下的核心业务主链路

当前共享底座默认采用以下主链路理解：

初始化 / 登录 / 进入桌面  
→ 识别和管理磁盘  
→ 创建和管理存储池、卷等存储对象  
→ 通过 Web 管理端及其他端进行查看、配置、访问  
→ 在异常、告警、状态变化、容量变化、删除 / 迁移 / 修复等场景下系统仍能正确表现

这条主链路意味着：

- 登录和进入系统不是孤立入口，而是主链路起点
- 磁盘、存储池、卷等对象不是普通数据项，而是存储主链路中的核心对象
- 查看、配置、访问都要建立在对象和状态正确的基础上
- 异常、失败、边界和恢复场景不是附属验证，而是主链路的一部分

## 7. 为什么测试时必须关注多端、多对象、多状态一致性

测试 TNAS 时，如果只看“某个页面能不能点”，很容易漏掉真正关键的问题。  
必须额外关注以下一致性：

### 多端一致性

- Web 端与手机端对同一对象的展示是否一致
- 不同端是否对同一操作给出一致结果
- 一端状态变化后，另一端是否正确同步

### 多对象一致性

- 磁盘、存储池、卷等对象之间关系是否正确
- 对象之间是否在展示、操作和结果上保持逻辑一致

### 多状态一致性

- 创建中、删除中、迁移中、修复中、异常、失败等状态是否合理
- 过程态与最终态是否一致
- 页面展示状态是否和真实业务状态一致

### 前后端一致性

- 页面显示是否与接口返回一致
- 页面状态是否与后台真实状态一致
- 不同页面、不同模块之间展示口径是否一致

### 文档与实现一致性

- 接口文档是否与真实请求 / 响应一致
- 字段描述是否与实际含义一致
- 文档描述是否能支撑真实测试判断

## 8. 为什么不能只看页面能不能点

在 TNAS 语境里，只看页面是否可点，会遗漏很多高风险问题，例如：

- 页面展示成功，但后台对象状态错误
- 页面状态更新了，但真实接口数据未更新
- 不同模块看到的同一对象信息不一致
- 文档写法正确，但真实请求缺字段或字段含义不同
- 异常和失败场景没有正确反馈
- 手机端与 Web 端状态不同步

因此后续所有测试分析默认遵循一个前提：

**页面现象只是入口，不是结论。**  
真正的判断要结合对象、状态、接口、日志、文档和业务规则一起完成。

## 9. 当前业务总览中的待补充点

以下内容后续建议在第二阶段继续补充，但当前不影响第一阶段底座成立：

- 设备端、PC 端或其他客户端能力的具体测试边界
- 更细的业务角色模型和权限边界

## 10. 第一批知识库补充的产品定义口径

第一批知识库进一步给出了更接近产品原始定义的描述：

- TNAS 全称为 `TerraMaster Network Attached Storage`
- 本质上是一个可以集中存储照片、影片、音乐及办公文件等重要数据的网络存储装置
- 它既可以作为个人使用，也可以作为亲友、同事共享的存储空间
- 用户既可以通过计算机访问，也可以通过移动 App 在手机等移动设备上访问
- 在测试语境里，可以把 TNAS 理解成“专属私有云盘”体系

这与当前共享底座中“私有云存储设备体系”的理解是相互印证的。

## 11. 第一批知识库补充的产品形态与机型视角

第一批知识库进一步说明，TNAS 的产品形态并不是单一设备，而是包含多种产品系列和机型组合。  
当前共享层先吸收以下稳定认知：

- 产品代号通常由产品类型字母、盘位数、核数和迭代数组成
- 当前知识库中已出现的产品类型包括：
  - `F 系列`：中小型塔式 NAS
  - `T 系列`：专业级 NAS
  - `U 系列`：机架式
  - `D 系列`：直连 DAS

这进一步说明，后续测试不能把 TNAS 理解成单一型号的软件验证，而应带着“多机型、多架构、多硬件差异”的意识去理解测试边界。

## 12. 第一批知识库补充的架构视角

第一批知识库明确提到，TNAS 当前至少涉及两类处理器架构：

- `ARM`
- `x86`

当前共享层吸收的稳定认知包括：

- ARM 架构更偏低功耗、成本敏感场景
- x86 架构更偏高性能、并发能力和复杂应用场景
- 不同架构下，安装能力、兼容性、性能表现和测试重点可能不同

这意味着后续分析某个问题时，应保留“是否与机型或架构有关”的判断意识。

## 13. 第一批知识库补充的初始化与安装测试口径

第一批知识库已经补充了一批对测试很关键的初始化和安装事实，当前共享层先吸收以下稳定口径：

- 安装 TOS 时会对新硬盘进行固定分区，至少包括：
  - 引导分区
  - 系统分区
  - 交换分区
  - 数据分区
- 不同 TOS 版本、不同架构下，系统分区大小和引导方式可能存在差异
- 自动安装和自定义安装的行为存在明显差异
- TOS 6 自动安装场景下，如果全部是新盘，会自动创建 `TRAID`
- 如果硬盘已有存储池信息，则不会重复自动创建
- 自定义安装场景下，不会自动创建 `TRAID`，需要进入系统后手动创建
- 当前知识库中明确提到，最新版本 Bootloader 最多只能选择 `2 块硬盘` 作为系统盘
- 当前知识库中明确提到，相关安装场景只支持安装 `TOS 6` 及以上版本，不能推荐用户倒退版本使用

这些内容对后续以下测试方向有直接价值：

- 初始化测试
- 安装测试
- 升级 / 重装测试
- 机型兼容性测试
- 存储主链路测试

## 14. 当前对产品总览的使用边界

虽然第一批知识库已经提供了大量机型、处理器和引导细节，但当前共享层只吸收“稳定且通用的产品理解”。  
以下内容仍建议保留到第二阶段按专题或模块继续展开：

- 各系列具体机型清单
- 各机型启动方式差异
- 各版本分区大小细则
- 各架构和具体处理器的专项兼容性规则
