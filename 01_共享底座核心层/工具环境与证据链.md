# 工具环境与证据链

## 1. 当前测试对象

当前测试对象明确包括：

- Web 端
- 手机端

这意味着后续测试不能默认只围绕 Web 端展开。  
任何需要解释现象、判断问题或设计验证路径的协作，都应把 Web 端和手机端都纳入当前工具语境。

## 2. Web 端相关工具与观察方式

当前 Web 端测试和定位问题时，主要会结合以下工具和方式：

- 浏览器开发者工具
- Fiddler

其中：

- 浏览器开发者工具主要用于页面行为、网络请求、页面调试信息等基础观察
- Fiddler 主要用于抓包、请求 / 响应分析、请求链路核对和问题还原

因此，Web 端问题判断不应只看页面表象，还应结合浏览器观察和抓包结果一起完成。

## 3. 手机端相关工具与观察方式

当前手机端抓包方式为：

- Fiddler + MuMu 模拟器

手机端问题定位时，还会结合：

- 模拟器日志

这意味着手机端问题不能只凭页面现象提结论，通常还需要结合：

- 抓包请求 / 响应
- 模拟器日志
- 页面现象
- 复现步骤

## 4. 接口测试常用工具

当前接口测试常用工具包括：

- Apifox
- Postman
- JMeter

这些工具主要服务于：

- 接口验证
- 参数与返回校验
- 链路验证
- 重复性验证
- 性能或批量接口场景的辅助检查

说明：

- 第一阶段共享底座只确认工具清单和使用方向
- 具体到各模块接口规则、接口模板和数据策略，后续可继续拆分

## 5. 自动化相关环境

当前自动化能力属于测试能力的一部分，主要服务于：

- 接口验证
- 重复性回归
- 提效
- 链路验证

与自动化配套的常用环境和工具包括：

- Windows
- VS Code
- PyCharm
- Git

这里需要明确：

- 自动化不是共享底座的唯一中心
- 当前共享底座对自动化的定位，是为测试工作服务，而不是默认切换到开发实现模式

## 6. 其他常用环境与工具

当前常用环境和工具还包括：

- Chrome
- Markdown
- Obsidian
- SSH / Linux 命令

这些工具在当前工作中的典型作用包括：

- 浏览器端观察和验证
- 文档记录和沉淀
- 知识整理
- 必要的信息查看和辅助判断

## 7. 当前可观测能力

当前问题判断不会只盯页面本身，而是会综合看：

- 页面现象
- 接口请求 / 响应
- Cookie / Token / Header 等认证信息
- 抓包结果
- 模拟器日志
- 真实操作链路
- 必要时的设备 / 系统层信息

这说明当前可观测能力不是单点式的，而是跨页面、接口、日志和系统信息的综合观察。

## 8. 当前最核心的证据链

当前最依赖的证据链是：

页面现象  
+ 抓包请求 / 响应  
+ 接口文档 / 需求文档  
+ 模拟器日志或相关系统信息  
+ 缺陷复现步骤  
+ 最终结论

这条证据链体现出一个核心原则：

**测试结论不是单一来源给出的，而是多种真实证据交叉判断后的结果。**

## 9. 为什么结论依赖多源证据交叉验证

单看页面现象，容易出现以下误判：

- 页面显示异常，但根因其实是接口字段或状态口径问题
- 页面显示正常，但后台对象状态其实错误
- 文档描述看似没问题，但真实抓包并不一致
- 手机端与 Web 端对同一状态的展示不同

因此，当前共享底座明确采用多源证据交叉验证原则：

- 页面现象提供问题入口
- 抓包请求 / 响应提供行为还原
- 文档提供规则参考
- 日志和系统信息提供辅助佐证
- 复现步骤提供问题闭环路径

只有把这些证据连起来，结论才足够可靠。

## 10. 工具与测试类型的对应关系

| 测试类型 | 常用工具 / 方式 | 主要作用 |
| --- | --- | --- |
| 功能测试 | Web 页面、手机端页面、浏览器开发者工具 | 验证页面、链路、交互、状态 |
| 接口测试 | Apifox、Postman、JMeter | 验证接口、参数、返回、链路 |
| Web 抓包 | Fiddler、浏览器开发者工具 | 还原真实请求、分析页面与接口关系 |
| 手机端抓包 | Fiddler + MuMu 模拟器 | 分析手机端请求与异常现象 |
| 手机端定位 | 模拟器日志 | 辅助定位和提 Bug |
| 文档沉淀 | Markdown、Obsidian | 记录、归档、长期复用 |
| 辅助查看 | SSH / Linux 命令 | 必要信息查看和辅助判断 |

## 11. 当前维护原则

维护本文件时默认遵守以下原则：

- 工具清单只记录当前真实在用的工具
- 证据链优先记录真实做法，不写理想化流程
- 如果某类工具未来变化，应同步更新工具清单和证据链描述
- 结论必须体现多源证据交叉验证思路，不允许退化为单点判断

## 14. 关于内网资料位置的处理原则

第一批知识库中已经出现了公司知识库、综合资料、应用包和测试素材的内网位置。  
当前共享底座对这类信息采用以下处理原则：

- 知道其存在并承认其是当前资料体系的一部分
- 不在核心共享层里大面积复制具体内网路径
- 需要时优先回到原始知识库文档查阅

这样做的目的，是避免核心共享层被大量路径信息淹没，同时保留资料可追溯性。
