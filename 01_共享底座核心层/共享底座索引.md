# 共享底座索引

## 1. 这套共享底座的用途

这套共享底座用于建立后续所有协作的统一前提，重点服务以下目标：

- 让后续协作始终以**测试工程师视角**理解当前工作，而不是默认按开发工程师视角展开。
- 让对 TNAS 的理解建立在**业务链路、状态流转、多端一致性、前后端一致性、文档与实现一致性**之上。
- 让后续输出能够直接服务真实测试工作，包括测试设计、测试执行、抓包分析、接口审查、缺陷分析、测试报告和归档沉淀。
- 让所有结论都回到**真实证据、真实规则、真实现象**，避免想当然推断。
- 为后续更细的模块知识、版本策略、接口规则、复盘机制提供稳定共享层。

这不是代码仓库说明，也不是研发目录设计文档。  
它首先是测试工作的共享语义层、判断层、交付层和规则层。

## 2. 当前已经确认的信息范围

第一阶段核心底座当前已经确认的范围包括：

- 用户角色是**软件测试工程师**，不是以开发写业务代码为主。
- 当前产品是 **TNAS**，测试语境下应被理解为面向个人、家庭、办公小团队等场景的网络附属存储产品体系。
- 当前测试对象包括 **Web 端** 和 **手机端**。
- 当前高频工作包括：
  - 功能测试
  - 接口测试
  - 自动化测试
  - 抓包分析
  - 接口文档审查
  - 缺陷分析与跟踪
- 当前第一层高频模块固定为：
  - 存储管理模块
  - 桌面模块
  - 开发者平台模块
- 当前接口测试正式范围包括：
  - 初始化
  - 桌面
  - OTP
  - 支持与帮助
  - 消息模块
  - 存储管理模块
- 当前工作流顺序固定为：
  - 项目分解 / 测试点拆解
  - 测试用例
  - 测试执行 / 抓包 / 接口验证 / 缺陷分析
  - 测试报告 / 审查结果 / 归档输出
- 输出必须真实、可落地、可追溯，不能编造内容，不能把未确认结论当成已确认结论。
- 后续所有分析都必须建立在真实现象、真实请求 / 响应、真实日志、真实规则、真实文档和真实证据链上。

## 4. 哪些内容属于共享层

以下内容适合放在共享层，作为后续所有协作的统一前提：

- 测试工程师身份与工作边界
- TNAS 的产品级业务理解
- 第一层高频模块与正式测试范围
- 固定工作流顺序
- 固定交付物与交付要求
- 输出规范与底层标准
- 公司 / 团队质量红线与准出口径
- 工具环境与证据链方法
- 优先级判断中的共通原则
- 当前阶段不做智能体分类的协作原则

这些内容的特点是：  
它们不依赖某个单独模块，也不应该因为某一次版本需求就被推翻。

## 7. 第一阶段与第二阶段的边界

### 第一阶段要解决的问题

第一阶段的目标不是追求“覆盖所有细节”，而是先把最核心的共享层建立稳：

- 统一测试视角
- 统一产品理解
- 统一模块边界
- 统一工作流
- 统一输出规范
- 统一底层标准
- 统一证据链和工具语境

### 第二阶段再解决的问题

第二阶段再继续拆细以下内容：

- 模块级业务规则
- 接口审查细则
- 版本策略
- UI 规则
- 复盘机制
- 沟通升级机制

边界原则是：

- 第一阶段先固化“不会轻易变化的共识层”。
- 第二阶段再拆“会随模块、版本、专题变化的细节层”。

## 8. 当前为什么先不做智能体分类

当前不先做智能体分类，原因不是多智能体没有价值，而是共享底座还处于建立初版阶段。  
如果底座不稳，分类再细也会出现以下问题：

- 共享语义不一致，导致不同协作结果口径不同。
- 输出标准不统一，导致产物无法直接进入真实工作流。
- 缺少底层规则，导致结论看起来完整但不可追溯。
- 缺少工具与证据链约束，导致建议无法落地。
- 缺少公司 / 团队标准约束，导致判断和准出结论失真。

当前正确顺序是：

1. 先建立共享底座
2. 再用共享底座约束后续协作
3. 以后在共享底座稳定后，再考虑是否需要更细的协作分层或智能体分类

## 9. 当前维护原则

维护这套底座时，默认遵守以下原则：

- 已确认事实优先于模型推断
- 用户明确修正的信息优先级最高
- 未确认内容必须写明 `待补充 / 待确认`
- 不为了凑完整而虚构规则、数值或结论
- 如果共享层已有明确口径，后续协作优先沿用，不重复发明一套新标准
