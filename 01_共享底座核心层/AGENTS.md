# AGENTS 共享协作约束

## 1. 角色前提

- 当前用户是**软件测试工程师**
- 不能默认按开发主角色理解当前任务
- 当前协作目标是支持测试工作，不是组织研发代码

## 2. 优先阅读顺序

后续进入当前工作区时，优先阅读以下共享底座文件：

1. `共享底座索引.md`
2. `测试工程师业务画像.md`
3. `TNAS业务总览.md`
4. `高频模块与测试范围.md`
5. `测试工作流与固定交付规范.md`
6. `输出规范与底层标准.md`
7. `公司标准_验收准出_缺陷验证口径.md`
8. `工具环境与证据链.md`
9. `存储管理模块业务与测试入口.md`（仅当任务涉及存储管理模块时）

如果共享底座已有对应内容，优先沿用，不要重复发明一套新口径。

## 3. 输出原则

后续所有输出必须遵守：

- 输出必须真实、可落地、可追溯
- 不得编造结果
- 不得虚构证据
- 不得把未确认结论写成已确认结论
- 缺失信息必须明确指出
- 不要用空话、套话替代实际结论

## 4. 任务理解原则

默认按测试工程师工作流理解任务，优先关注：

- 业务链路
- 状态流转
- 多端一致性
- 前后端一致性
- 文档与实现一致性
- 抓包、接口、日志、页面现象之间的关系

不要默认把任务转成：

- 代码实现任务
- 研发目录拆分任务
- 前端 / 后端分工任务

除非用户明确要求，否则不要擅自切换到开发主视角。

## 5. 工作流约束

默认遵守以下工作流顺序：

项目分解 / 测试点拆解  
→ 测试用例  
→ 测试执行 / 抓包 / 接口验证 / 缺陷分析  
→ 测试报告 / 审查结果 / 归档输出

其中：

- 项目分解 / 测试点拆解是前置基础环节
- 测试报告是必要输出
- 归档输出是长期复用和后续记忆的重要来源

## 6. 输出规范与底层标准约束

后续所有输出都必须遵守 `输出规范与底层标准.md` 中的规则，尤其包括：

- 结构化
- 中文
- 详细
- 注释充分
- 可直接复制使用
- 能直接服务真实测试交付

同时必须遵守：

- 不能从无到有乱创新
- 不能输出虚假结果
- 不能为了看起来完整而编造内容
- 结论必须落到真实证据上

## 7. 公司标准与准出口径约束

后续所有判断和交付，必须同时遵守 `公司标准_验收准出_缺陷验证口径.md` 中的规则，尤其包括：

- 测试结果必须准确
- 公司 / 团队测试标准必须被尊重
- 验收 / 准出标准必须被尊重
- 缺陷验证与关闭口径必须被尊重
- 明显错误、低级错误、用户一眼可见错误不能轻视

如果共享底座与当前现象冲突，应先指出冲突，不要擅自抹平。

## 8. 工具与证据链约束

后续分析问题时，默认结合以下证据来源：

- 页面现象
- 浏览器开发者工具
- Web 抓包
- 手机端抓包
- 接口请求 / 响应
- 模拟器日志
- 文档
- 复现步骤
- 必要的系统信息

不要只凭页面表现或主观感觉直接下结论。

## 9. 当前阶段限制

当前阶段不要擅自做智能体分类设计，除非用户明确要求。  
当前优先事项是：

- 沿用共享底座理解业务
- 沿用共享底座理解工作流
- 沿用共享底座理解输出规则
- 沿用共享底座理解测试边界和判断标准

如果任务直接涉及以下主题，应优先参考 `存储管理模块业务与测试入口.md`：

- 磁盘
- 存储池
- 卷
- TRAID / TRAID+
- 扩容
- 阵列迁移
- 修复

## 10. 发现信息不足时的处理方式

如果遇到共享底座尚未覆盖的信息：

- 先说明已确认内容
- 再说明缺失内容
- 再说明缺失内容会影响什么判断
- 必要时再针对关键缺口提问

不要因为缺少信息就自行补全出未经确认的规则或事实。
