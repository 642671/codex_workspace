# 输出规范与底层标准

## 1. 这份文件解决什么问题

这份文件用于统一后续所有输出的基本要求。  
它不讨论某个具体模块，而是定义：

- 什么样的输出才算真的能用
- 什么样的输出不能接受
- 后续交付应遵守哪些底层标准
- 为什么验收标准 / 实收标准必须被当作更底层判断依据

## 2. 当前偏好的输出形式

当前偏好的输出形式包括：

- Markdown
- 表格
- 可直接复制到 Excel 的文本表格
- XMind
- 测试平台 / 缺陷系统中的正式记录

这意味着后续输出不应只追求“解释得通”，还应追求：

- 便于直接复制
- 便于直接评审
- 便于直接提单
- 便于直接归档
- 便于直接进入正式系统

## 3. “能直接拿去用”的标准

当前默认采用以下标准判断内容是否“能直接拿去用”：

- 中文
- 结构化
- 详细
- 注释充分
- 逻辑清晰
- 可直接复制使用
- 可直接进入测试、评审、提单、归档流程
- 不空泛
- 不只讲道理不落地

如果一份内容只是“听起来完整”，但不能直接进入真实工作流，就不算合格输出。

## 4. 结构化要求

后续输出应尽量具备以下结构特征：

- 有明确标题
- 有清晰分段
- 有稳定字段或维度
- 有结论和依据
- 有风险和待确认项
- 有必要时的证据说明

如果输出对象适合表格表达，应优先使用结构清晰的表格或可复制表格文本。  
如果输出对象适合层级表达，应优先使用 XMind 或树状结构。  
如果输出对象适合长期沉淀，应优先使用 Markdown。

## 5. 详细、注释充分、可复制使用的要求

“详细”不等于堆字数，而是要把真正影响执行和判断的细节写清楚。  
当前默认要求：

- 关键背景不能缺
- 关键条件不能缺
- 关键步骤不能缺
- 关键字段不能缺
- 关键结论不能缺
- 关键风险不能缺
- 必要时要补充注释说明

“可复制使用”意味着内容不是只能阅读，还能直接拿去做事，例如：

- 直接复制进 Excel
- 直接复制进缺陷系统
- 直接作为评审材料
- 直接进入测试记录
- 直接作为归档文档

## 6. 禁止事项

后续所有输出必须遵守以下禁止事项：

- 不能从无到有乱创新
- 不能输出虚假结果
- 不能为了看起来完整而编造内容
- 不能把未确认结论当作已确认结论
- 不能用空话、套话代替实际结论
- 不能只给原则，不给能落地的内容
- 不能回避关键缺失信息

如果当前信息不够，就应明确写出：

- 已确认内容
- 缺失内容
- 缺失内容会影响什么判断

## 7. 真实证据要求

后续输出中的结论必须落到真实证据上。  
可作为结论支撑的典型证据包括：

- 页面现象
- 真实请求 / 响应
- 抓包结果
- 模拟器日志
- 相关系统信息
- 需求文档
- 接口文档
- 缺陷复现步骤
- 已确认规则

没有证据链的内容，最多只能算推测或待确认点，不能包装成既定结论。

## 8. 缺失信息必须明确指出

如果当前信息不完整，输出中必须明确标注：

- `待补充`
- `待确认`
- `缺少原始证据`
- `缺少规则依据`
- `缺少环境信息`

这不是“不完整”，而是对真实情况的尊重。  
共享底座明确反对为了好看而掩盖信息缺口。

## 9. 验收标准 / 实收标准属于更底层判断依据

当前共享底座明确把以下内容视为更底层判断依据：

- 验收标准
- 实收标准
- 团队已有测试标准
- 实际业务结果
- 真实请求 / 响应 / 抓包 / 日志 / 页面现象

这类标准不是“个人偏好”，而是更接近测试判断底线的标准层。  
后续所有测试设计、执行、结论、准出和缺陷处理，都应优先服从这些标准。

## 10. 输出必须服务真实测试交付

当前共享底座默认要求：  
所有输出都必须能服务真实测试交付，而不是只服务一次性讨论。

这意味着输出至少要能支撑以下一种或多种真实动作：

- 测试执行
- 测试评审
- 接口验证
- 抓包分析
- 缺陷提单
- 缺陷验证
- 测试报告编写
- 验收 / 准出判断
- 归档沉淀

## 11. 推荐输出骨架

后续遇到需要结构化输出的场景，建议优先采用如下骨架之一：

### 骨架一：结论型

- 背景
- 已确认事实
- 关键结论
- 证据依据
- 风险点
- 待确认项

### 骨架二：测试执行型

- 测试范围
- 前置条件
- 操作步骤 / 验证点
- 实际结果
- 预期结果
- 证据说明
- 问题记录

### 骨架三：审查型

- 审查对象
- 已发现问题
- 问题类型
- 影响说明
- 证据或对比依据
- 建议处理

## 12. 当前维护原则

维护本文件时默认遵守以下原则：

- 先保证真实，再追求完整
- 先保证可落地，再追求形式漂亮
- 先说明依据，再输出结论
- 如果已有更底层标准，以更底层标准为准

## 13. 第一批知识库补充的输出口径

第一批知识库进一步确认了几个会直接影响真实交付的输出要求：

- 测试结果必须准确，避免影响项目决策
- 对于测试不深入或发生重构的功能，测试人员应主动梳理测试要点并发起评审
- 对不明确的功能点，必须先与产品经理确认，不能自行假定
- 提交缺陷时，应尽可能注明可复现概率

这些要求说明，输出不仅要“有内容”，还要具备以下特征：

- 能用于决策
- 能用于复查
- 能用于跨角色协作
- 能把不确定性显式表达出来

## 14. 当前对“准确输出”的进一步理解

结合第一批知识库，当前共享底座对“输出准确”的理解进一步细化为：

- 不能把未验证结论当成结果汇报
- 不能把局部验证结果包装成完整测试结论
- 不能遗漏影响发布、验收或缺陷关闭的关键条件
- 不能在缺少规则依据时强行给出肯定结论

因此，后续只要涉及：

- 测试报告
- 缺陷单
- 审查结果
- 归档结论

都必须明确区分：

- 已验证
- 已观察
- 已确认规则
- 仍待确认

## 15. 缺陷单最小信息要求

根据第一批吸收的缺陷流程表，当前共享底座进一步明确：  
后续缺陷输出不是“写个标题加截图”就够，而应至少具备以下最小信息：

- 标题
- 是否可复现
- 做了什么
- 得到的结果
- 期望结果
- 前端 / 后端错误日志
- 复现环境或保留环境说明
- 当前测试版本号
- 对应模块
- 优先级
- 严重程度
- 处理人
- 预计开始时间
- 预计结束时间

其中几个关键点必须特别强调：

- 应用类缺陷需要写清楚具体应用版本和系统版本
- 需要写清楚基本操作步骤，不能只放一张截图
- 得到的结果必须文字化描述，不能只放截图
- 如果期望结果需要产品确认，应明确写成“待产品确定期望结果”
- 难复现问题需要尽量保留复现环境

## 16. 缺陷标题标签规范

根据缺陷流程表，当前共享底座已确认以下常用标题标签规则：

- `【用户反馈】`
- `【TNAS Mobile X.X.X】`
- `【回归缺陷】`
- `【语言库】`
- `【需求类】`
- `【优化类】`
- `【无法复现】`
- `【延期解决】`
- `【需求变更】`
- `【已转需求】`

这些标签不是装饰，而是有实际流程意义。  
后续输出缺陷分析或归档时，如果涉及这些场景，应优先沿用已有标签口径。

## 17. 可复现性表达要求

根据缺陷流程表，当前共享底座进一步确认：

- 缺陷必须区分 `可以`、`不可以`、`偶尔复现`
- 对偶尔复现的问题，应补充复现概率、复现条件和分析
- 对无法复现的问题，处理人不能直接填研发

这和此前“尽可能注明可复现概率”的规则相互印证。  
因此后续所有缺陷相关输出，都应主动体现可复现性信息，而不是只写“已复现 / 未复现”。
