# 公司标准、验收准出与缺陷验证口径

## 1. 这份文件的定位

这份文件用于承接共享底座中更底层的规则层。  
它不只是个人偏好记录，而是用于统一后续以下判断：

- 测试结果是否能被正式采信
- 版本是否具备验收 / 准出条件
- 缺陷应如何验证、关闭或升级处理
- 哪些问题不能被轻视
- 哪些情况必须升级汇报

说明：

- 当前文件中的规则分为两类：  
  - 已明确的协作默认口径  
  - `待补充 / 待确认` 的公司或团队细则
- 若后续获得公司正式制度文本，应以正式制度为准，并回写本文件

## 2. 当前默认遵循的质量底线

当前共享底座默认遵循以下质量底线：

- 测试结果必须准确
- 输出结论必须有依据
- 不能输出虚假结果
- 不能为了推进流程而跳过关键验证
- 不能把未确认结果包装成已验证通过
- 对明显错误、低级错误、用户一眼可见错误不能轻视

这些属于当前阶段的基础质量底线，后续协作默认必须遵守。

## 3. 公司 / 团队测试标准

当前已明确需要将“公司 / 团队测试标准”作为共享底座的一部分。  
第一阶段先固化以下默认口径：

- 所有正式测试结论必须建立在真实执行和真实证据上
- 测试过程应保留必要记录
- 关键问题应可追溯到现象、步骤、证据和判断依据
- 版本结论不能脱离测试范围、风险说明和遗留问题独立存在

`待补充 / 待确认`：

- 公司正式测试制度文件名称
- 团队现行测试标准的完整条文
- 是否存在统一模板、统一签字流或统一看板要求

## 4. 质量红线

当前共享底座先定义以下质量红线：

- 影响核心业务主链路的问题不能被忽略
- 影响数据安全或存储核心能力的问题不能轻视
- 会造成状态流转错误或对象关系错误的问题不能轻视
- 阻塞测试、阻塞发布的问题必须明确暴露
- 明显错误、低级错误、用户一眼就能看到的问题不能因为“看起来简单”而降级处理
- 前后端不一致、文档与实现严重不一致的问题不能直接带过

`待补充 / 待确认`：

- 公司是否有明确的质量红线清单
- 是否存在固定的 P0 / P1 级红线定义

## 5. 正式版本的测试完成要求

当前共享底座按以下口径记录正式版本要求：

- 正式版本必须完整执行 QA 用例并签字
- 测试范围、测试结果、遗留问题和风险说明必须完整留痕
- 不能在缺少关键验证的情况下直接给出“通过”结论

说明：

- “签字”当前先按正式确认动作理解，可落在系统、文档或流程中
- 具体签字主体、签字形式、适用版本范围，后续需要按团队现行流程补充

`待补充 / 待确认`：

- 签字人角色与责任边界
- 使用的签字载体或系统
- 适用范围是全部正式版本还是特定版本类型

## 6. 验收 / 准出标准

当前共享底座先固化以下准出判断框架：

- 测试范围已明确
- 关键测试已完成
- 高风险问题已暴露并形成处理结论
- 缺陷状态已达到当前版本允许发布的口径
- 测试报告已完成
- 相关文档和记录已具备基本完整性

关于用户明确要求纳入的几项内容，当前默认口径如下：

### Bug 修复率

- Bug 修复率应满足当前版本准出要求
- 具体数值阈值 `待补充 / 待确认`

### 回归要求

- 修复后的问题必须完成必要回归
- 影响范围内的相关链路应完成对应验证

### 测试报告要求

- 测试报告是必要输出
- 没有测试报告，不应默认视为本轮测试正式完成

### 文档完整性要求

- 测试相关文档、审查记录、缺陷记录、归档材料应具备基本完整性
- 具体完整性清单 `待补充 / 待确认`

## 7. 缺陷验证前检查内容

当前默认缺陷验证前应至少检查以下内容：

- 当前验证版本 / 构建信息是否正确
- 验证环境是否正确
- 缺陷复现步骤是否完整
- 前置条件是否满足
- 修复范围是否明确
- 相关功能是否已正确部署
- 是否需要依赖接口、账号、数据或特定配置
- 是否有可对比的页面现象、抓包、日志、文档或历史记录

如果这些内容缺失，应明确说明缺口，而不是直接下“已验证”或“无法复现”结论。

## 8. 缺陷关闭标准

当前共享底座默认采用以下缺陷关闭口径：

- 缺陷对应问题已按复现路径验证通过
- 实际结果与预期结果一致
- 关键证据链闭合
- 必要的相关回归已完成
- 不存在明显同类回归风险未验证的情况
- 缺陷结论可追溯

说明：

- “关闭”不是简单改状态，而是建立在真实验证基础上的结果
- 若修复与预期不一致、验证条件不足或风险未说明，不应直接关闭

## 9. 延期缺陷、疑难缺陷、无法复现缺陷处理口径

### 延期缺陷

当前默认要求：

- 必须写明延期原因
- 必须写明风险影响
- 必须明确计划处理版本或后续跟踪方式
- 必须说明是否影响当前准出

### 疑难缺陷

当前默认要求：

- 必须保留现象、条件、证据、排查轨迹
- 必须明确当前卡点在哪里
- 必须明确是否需要跨角色协同处理
- 必须判断是否需要升级汇报

### 无法复现缺陷

当前默认要求：

- 不能只写“未复现”就结束
- 必须写明验证环境、验证版本、验证步骤、验证次数和现象
- 必须说明当前证据是否不足
- 必须说明是否需要继续观察、回归或补充材料

## 10. 需要升级汇报的触发条件

当前共享底座建议将以下情况视为应升级汇报的触发条件：

- 影响核心业务主链路
- 影响数据安全或存储核心能力
- 阻塞测试或阻塞发布
- 多用户受影响
- 可能引发用户客诉
- 明显错误或低级错误出现在正式提测或待发布版本中
- 同类问题反复出现或反复 reopen
- 规则冲突导致无法判定是否可发布
- 缺陷长期无法定位、无法稳定复现或风险持续扩大

`待补充 / 待确认`：

- 当前团队的正式升级路径
- 升级接收角色
- 必须同步的群组、会议或平台

## 11. 关于明显错误、低级错误、用户一眼可见错误

当前共享底座明确采用以下口径：

- 明显错误不能发生
- 低级错误不能发生
- 用户一眼就能看到的问题不能被轻视

不能只按“是否影响数据”或“是否阻塞发布”来判断问题轻重。  
因为这类问题直接影响：

- 用户感知
- 版本观感
- 产品专业度
- 测试质量可信度

## 12. 第一阶段仍待补充的制度细节

以下内容第二步应尽快补齐，但第一阶段先不擅自编具体制度值：

- 公司正式测试标准原文或摘要
- 质量红线的正式分级标准
- Bug 修复率具体阈值
- 正式版本签字的具体流程与载体
- 验收 / 准出的正式模板和审批口径
- 无法复现、延期缺陷、疑难缺陷的公司制式状态流转规则

## 13. 第一批知识库已确认的缺陷管理细则

第一批知识库已经明确了以下缺陷管理细则，应正式纳入共享底座：

### 缺陷提出前提

- 提出缺陷前，测试人员需确认自己已足够理解需求
- 缺陷报告应建立在合理测试判断和真实现象之上

### 缺陷时效

- Bug 周期从分配给研发工程师或产品经理开始计算
- 当前已确认的默认缺陷周期为 `5 个工作日`
- 超时未解决的缺陷必须向上一级领导汇报
- 对于几周未解决且未开会讨论的缺陷，不能默认挂起，必须推动暴露和处理

### 验证时效

- 已修复缺陷应尽快验证
- 非重大事项原则上一周内完成验证

### 延期标识

- 复杂缺陷可安排更长时间，例如两周
- 但必须在进度统计或缺陷标题中标注清楚，例如 `【延期解决】`

### 缺陷评论与原因记录

- 缺陷验证前必须检查评论中是否已写明具体原因和解决对策
- 每次修改缺陷状态或流转状态时，测试工程师必须进行评论
- 讨论后确认有效的常见问题解决方法，必须录入团队知识库
- 反复出现的问题应立项专项跟进

### 临时对策处理

- 如果研发给出的只是临时对策，不能直接视为彻底解决
- 必须监督研发工程师通过邮件向上级说明

## 14. 第一批知识库已确认的紧急缺陷与升级汇报口径

第一批知识库已经给出了更明确的紧急缺陷定义：

- 所有用户客诉问题，必须第一时间处理
- 量产版本中直接影响版本发布的阻塞性缺陷，应视为最高优先级
- 多用户遇到并反馈的缺陷，一律视为紧急缺陷

因此当前共享底座可将以下场景直接视为高优先级升级对象：

- 用户客诉
- 发布阻塞
- 多用户反馈
- 长期未解决且未形成讨论机制的问题

此外，第一批知识库还明确要求：

- 对一直解决不了、难以复现或无法明确原因的 Bug，必须及时上报领导或寻找他人帮助
- 当任务冲突、资源不足、流程执行受阻或技术争议无法达成一致时，应及时升级汇报
- 对复杂耦合性问题，应召开专项会议深入讨论，而不是简单按单点问题处理

## 15. 第一批知识库已确认的发布铁律与准出补充规则

第一批知识库已将以下内容明确为正式版发布铁律：

- 最终发布的版本必须是经过 Beta 的系统包
- 所有发布的最终包都必须完整测试完 QA 用例，并获得签字确认
- 发布前应在公司内部和家庭环境进行测试
- 重要版本应进行灰度测试

第一批知识库还明确了正式版本的修改边界：

- 严格禁止往 TOS 正式版本中增加新功能
- 对正在使用且影响面大的功能，不能直接修改
- 如必须改动，需综合评估后打包大版本，并经过足够验证和 Beta 测试

第一批知识库还进一步强调：

- 正式版不能出现问题
- 出现品质事故后将按公司规定严厉处罚
- 知识库原文已给出绩效处罚示例：第一次 20%，第二次 50%，第三次 100%

当前共享底座将以上内容作为已确认的公司规则来源记录，后续如获得更正式制度文本，应再做正式对齐。

## 16. 第一批知识库已确认的环境、安全与线上变更红线

第一批知识库已明确以下红线：

- 所有生产环境一律不准用于测试
- 线上服务器不准用于测试
- 任何对线上服务器的业务修改，必须提前提交申请
- 部署测试验证通过后，需再次提交上线申请，并附测试结果
- 未经申请擅自修改线上业务，属于严重违规

同时，第一批知识库还明确了以下安全和合规底线：

- 禁止将账号、密码等敏感信息明文存储于文件或代码中
- 所有涉及网络通信的功能应具备证书并使用 HTTPS
- 应用端口需增加对 TLS 的支持
- 禁止未经允许录音、拍照或泄露公司内部敏感信息

这些内容已经足够进入第一阶段共享层。  
更细的安全规范、接口规范和文档审查规则，第二阶段再单独拆分。

## 17. 第一批知识库已确认的协作与跟踪规则

第一批知识库进一步补充了以下协作与跟踪规则：

- 实行每日 / 每周缺陷修复进度报告制度
- 测试部门需将每周解决的 Bug 与计划解决清单进行对比
- 当需求或 Bug 被拒绝但未写明明确原因时，部门间需互相提醒
- 对于研发表示因某些特性无法解决的 Bug，应通过 TAPD 评论记录，以便后续追踪
- 重要事项和文档需通过邮件发送，并确保关键人员参与或知悉

这些内容说明，缺陷管理并不止于提单和验证，还包括：

- 进度监督
- 原因留痕
- 跨部门提醒
- 正式升级与邮件留痕

## 18. 第一批知识库已确认的缺陷流转状态

根据缺陷管理流程表，当前共享底座已确认以下正式状态流转节点：

- 新
- 接受 / 处理
- 评估中
- 评估通过
- 已解决
- 已验证
- 已合并
- 已关闭
- 重新打开
- 已拒绝

这些状态当前应被视为团队正式流转语义，而不是临时口头说法。

## 19. 各状态的关键责任口径

根据缺陷流程表，当前共享底座进一步确认：

### 新

- 由测试提交
- 覆盖测试发现的问题和用户反馈的问题
- 提交后应同步检讨缺陷原因
- 如果暴露出用例不完善，应同步完善用例

### 接受 / 处理

- 由研发进入处理

### 评估中

- 由研发进入评估

### 评估通过

- 由研发组长或主管确认通过

### 已解决

- 由研发标记

### 已验证

- 由测试标记
- 点击“已验证”前，研发必须已填写详细的缺陷原因分析和解决对策
- 测试必须评论说明在哪个测试版本中验证通过

### 已合并

- 由研发主管标记

### 已关闭

- 由测试标记
- 点击“已关闭”前，必须评论说明在哪个准备发布的正式版本中验证通过
- 如果不是修复关闭，而是决议关闭，则必须有清晰评论说明拒绝修改的合理理由或会议决议，并能看出决定人

### 重新打开

- 可由测试或产品发起
- 重新打开前，必须评论说明在哪个版本验证未通过
- 必须确认用于验证的版本打包时间晚于缺陷状态变为“已解决”的时间

## 20. 已拒绝状态的细分口径

根据缺陷流程表，`已拒绝` 不是单一含义，而是带有明确原因分类。  
当前已确认的细分原因包括：

- 无需解决
- 已修改
- 延期解决
- 无法重现
- 外部原因
- 重复
- 设计如此
- 问题描述不清楚
- 需求变更
- 已转需求
- 挂起

这些原因后续应作为缺陷分析和归档时的标准分类参考。

## 21. 各类已拒绝原因的处理要求

### 无需解决

- 需要产品确认无需解决
- 必须有评论写明拒绝原因
- 满足条件后可以关闭
- 如果被拒绝但仍认为应解决，应向对应领导升级

### 已修改

- 如果新版本在没有任何改动的情况下已经无法再次复现，可以关闭
- 但如果有修改动作，就必须由研发填写原因和对策

### 延期解决

- 必须经过会议讨论确定延期
- 需要产品或研发评论延期原因
- 后续应改成 `重新打开`
- 标题前加 `【延期解决】`
- 为避免统计失真，可视情况临时将处理人置空

### 无法重现

- 测试确认无法再复现后可以关闭

### 外部原因

- 与系统无关的外部操作或条件导致的问题，可评论后关闭

### 重复

- 必须确认确实属于重复缺陷后才能关闭

### 设计如此

- 必须确认需求中确实设计如此后才能关闭

### 问题描述不清楚

- 需先补充和修改描述
- 然后将缺陷改为 `重新打开`
- 为处理人重新选择 `5 个工作日` 作为解决周期

### 需求变更 / 已转需求

- 需要将缺陷改为 `重新打开`
- 处理人可以为空
- 必须评论关联对应需求
- 需求完成后再关闭缺陷
- 标题前应分别加 `【需求变更】` 或 `【已转需求】`

### 挂起

- 当前流程表明确写到“暂时没有人拒绝选择这个理由”
- 因此当前共享底座不把“挂起”当作默认推荐处理方式

## 22. 优先级与严重程度的正式区分

根据缺陷流程表，当前共享底座进一步明确：

### 优先级

强调“什么时候修、修复紧迫性如何”，当前分为：

- 紧急
- 高
- 中
- 低
- 无关紧要

其中已确认的紧急场景包括：

- 用户反馈的问题，一律按紧急处理
- 高危漏洞
- 影响系统发布、影响面广的问题
- 影响其他功能进一步测试的问题

### 严重程度

强调“问题本身的破坏性”，当前分为：

- 致命
- 严重
- 一般
- 提示
- 建议

其中已确认的致命场景包括：

- 系统频繁崩溃或死机，需要硬重启
- 文件损坏、数据丢失、数据损毁
- 核心访问能力不可用，例如 SMB / NFS / FTP / WebDAV 无法访问
- 在线或手动更新功能无法使用

这说明后续判断问题时，不能把优先级和严重程度混写成一个维度。

## 23. 计划时间的默认填写口径

根据缺陷流程表，当前共享底座进一步确认：

- 上午发现的缺陷，预计开始时间选当天上午
- 下午发现的缺陷，预计开始时间选次日上午
- 如果次日不是工作日，则顺延到下一个工作日
- 预计结束时间默认从开始时间往后选择，共计 `5 个工作日`

这为缺陷时效和进度跟踪提供了更细的操作口径。
