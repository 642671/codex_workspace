# 测试工程师业务画像

## 1. 我的角色定位

我是**软件测试工程师**。  
我的核心职责不是开发业务代码，而是围绕产品质量建立判断、验证和交付能力。
[[]]
我当前工作的重点包括：

- 功能测试
- 接口测试
- 自动化测试
- 抓包分析
- 接口文档审查
- 缺陷分析与跟踪

这些工作共同服务于一个目标：  
基于真实业务、真实现象、真实证据，判断产品当前是否满足预期，识别风险并推动问题闭环。

## 2. 我的工作边界

我的工作边界应从测试工程师视角理解，而不是从开发实现视角理解。

我关注的是：

- 需求是否被正确实现
- 业务链路是否正确
- 状态流转是否正确
- 多端表现是否一致
- 前后端是否一致
- 文档与实现是否一致
- 异常场景是否处理合理
- 风险是否被及时识别和记录
- 问题是否被准确分析、追踪和关闭

我不是以“写业务代码、组织研发目录、拆实现模块”为主角色。  
即使使用自动化能力，也应理解为测试能力的一部分，而不是默认进入开发主线。

## 3. 我的核心测试类型

当前我的高频测试类型包括：

### 功能测试

重点关注：

- 页面展示是否与需求一致
- 主流程是否走通
- 页面交互是否合理
- 状态变化是否正确
- 前后台展示是否一致
- 页面是否存在明显错误

### 接口测试

重点关注：

- 请求方法
- URL
- 请求头
- 认证方式
- 参数完整性
- 返回结构和关键字段
- 业务码 / 状态码
- 前置依赖
- 链路串联
- 实际返回与预期是否一致

### 自动化测试

自动化在当前工作中是重要能力，但不是唯一核心。  
它主要服务于：

- 接口验证
- 重复性回归
- 提效
- 链路验证

### 抓包分析

重点关注：

- 页面操作对应的真实请求是什么
- 请求路径 / 方法 / Header / Cookie / Token / 参数是否合理
- 页面现象是否能被真实请求和响应解释
- 文档与真实抓包是否一致

### 接口文档审查

重点关注：

- 接口描述是否准确
- 字段是否歧义
- 字段是否缺失
- 参数描述是否缺失
- 文档是否符合接口规范
- 文档是否与真实抓包一致

### 缺陷分析与跟踪

重点关注：

- 问题是否可稳定复现
- 复现路径是否完整
- 影响范围是否明确
- 证据链是否充分
- 缺陷状态与处理建议是否合理
- 是否需要升级汇报

## 4. 我的协作目标

我当前建立共享底座和后续协作体系的目的，不是组织研发代码，而是更好支持测试工作：

- 更快理解业务
- 更快进入正确测试语境
- 更稳定地产出测试结论
- 更规范地输出测试资产
- 更好地沉淀长期复用知识
- 让后续协作先继承共享底座，而不是每次重新猜一遍

## 5. 我如何做判断

我做判断，不依赖想当然推断，也不接受“看起来像对”的结论。  
我的判断方式依赖以下事实基础：

- 真实页面现象
- 真实请求 / 响应
- 真实抓包结果
- 真实日志
- 真实业务规则
- 真实需求 / 文档
- 真实复现步骤
- 已确认的验收标准 / 实收标准

判断一个问题时，我更关注：

- 它是否影响核心业务链路
- 它是否影响状态流转
- 它是否影响前后端一致性
- 它是否影响用户直接感知
- 它是否属于明显错误或低级错误
- 它是否会影响版本发布、用户使用或数据安全

## 6. 我如何交付

我的交付不是泛泛地“给个结论”，而是要形成可直接进入真实工作流的产物。  
当前常见固定交付包括：

- 项目分解 / XMind 分解
- 功能测试用例
- 接口测试用例
- 抓包分析记录
- 接口文档审查结果
- 缺陷分析与跟踪内容
- 测试报告
- 必要的归档文档 / Markdown 记录

这些产物必须满足一个前提：  
它们不是看起来完整，而是要能真正进入测试、评审、提单、汇报、归档和复用流程。

## 7. 我为什么重视真实证据而不是想当然推断

测试工作的本质不是“说出一个听起来合理的解释”，而是：

- 用真实现象发现问题
- 用真实请求 / 响应还原问题
- 用真实日志或系统信息佐证问题
- 用真实规则判断问题
- 用真实结论服务后续处理

如果没有证据链支撑，再完整的结论也不可靠。  
因此后续所有输出必须遵守：

- 不能从无到有乱创新
- 不能输出虚假结果
- 不能为了完整而编造内容
- 不能把未确认结论当成已确认结论
- 缺失信息必须明确指出

## 8. 我的测试视角和开发视角的区别

### 测试视角下更关注的内容

- 业务是否真的能跑通
- 状态和结果是否真的正确
- 页面、接口、日志、文档是否彼此一致
- 异常场景是否合理
- 风险是否被识别
- 结论是否有证据
- 产物是否可直接交付

### 开发视角下常见但不是我当前主目标的内容

- 代码目录如何拆分
- 哪个模块谁实现
- 如何组织前后端实现
- 如何做技术分层

这些内容不是完全无关，但当前不是共享底座的中心。  
当前共享底座必须优先服务测试判断和测试交付，而不是服务代码实现。

## 9. 当前协作默认原则

后续任何协作默认遵守以下原则：

- 优先按测试工程师视角理解当前任务
- 优先按共享底座已有口径输出内容
- 如果信息不足，明确说明缺失点
- 先保证真实和可追溯，再追求完整和漂亮
- 如果已有标准与当前现象冲突，先记录冲突，再推动澄清，不擅自抹平
